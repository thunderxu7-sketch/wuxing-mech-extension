# 五行校准 下一阶段规划

> 更新时间：2026-04-08
> 结论先行：旧规划的 `P0-P2` 已经大体落地，但还没有达到“可以继续冲 P3”的状态。下一阶段不该继续堆远期功能，而应先补齐正确性、可观测性和增长闭环。

## 二、为什么现在不建议直接做旧 P3

旧 P3 里的三个方向，优先级并不一样：

| 项目 | 是否建议现在做 | 原因 |
|------|----------------|------|
| 10. 符图数字商品化 | 不建议现在做 | 还没有验证用户会不会持续打开、保存、分享，更没有支付闭环；现在做付费功能太早 |
| 11. 小程序版本 | 暂缓立项，但保留为平台分支 | 这是渠道决策，不是功能迭代。只有当“中文用户有兴趣，但扩展安装转化差”被数据证实时才值得投入 |
| 12. 内容运营矩阵 | 建议立即开始，但按轻量实验做 | 这是运营动作，不需要等待新功能开发完成；可直接用现有分享图开始验证内容吸引力 |

一句话判断：现在最大的瓶颈不是“功能不够多”，而是“还没有一个可测量、可复用、可迭代的获客和转化闭环”。

---

## 三、当前项目真正缺的东西

### 1. 正确性还没收口

- 当天运势缓存只按日期存，没有按用户签名区分；同一天重新校准出生信息时，可能继续读到旧结果。
- 出生日期输入仍是裸数字输入，非法日期和边界值行为没有被系统验证。
- 项目目前能构建，但 `lint` 仍失败，说明基础工程质量还没收干净。
- 没有自动化测试，算法、缓存、时辰映射这些核心逻辑没有保护网。

### 2. 变现链路还没打通

- 商品链接本质上还是搜索页，不是真正可归因、可结算的联盟推广链接。
- 商品池仍硬编码在 `algorithm.ts`，后续运营和替换成本高。
- 没有商品图片，没有点击后转化跟踪，没有办法优化点击率。

### 3. 增长数据还不可用

- 当前埋点只存在 `chrome.storage.local`，开发时可看，产品决策时没用。
- 还没有安装、打开、分享、商品点击、回访之间的完整漏斗。
- 分享图缺少明确的回流入口，用户即使转发了，也不一定能带回安装或访问。

### 4. 分发与对外信息不完整

- `README.md` 还是默认 Vite 模板，项目对外描述、安装方式、截图、定位都没整理。
- 还没有“发布、安装、分享、追踪”的完整对外路径。

## 四、新规划

### Phase A：先补齐产品正确性（1 周）

目标：把“今天算出来的结果是对的”这件事做扎实。

1. 修复每日缓存策略
   - 缓存键至少包含“日期 + 用户签名摘要”。
   - 重新校准后应立即刷新今日结果，而不是命中旧缓存。

2. 收口工程质量
   - 修掉当前 lint 错误。
   - 清理明显无用或历史遗留逻辑。
   - 统一版本号和发布元数据。

3. 为核心逻辑补测试
   - `calculateUserSignature`
   - `calculateFortune`
   - `getDailyTalisman`
   - `hourToShichen`
   - `getDailyCache` / `setDailyCache`

4. 补输入校验
   - 校验非法日期。
   - 评估是否改为“日期选择器 + 时辰下拉”。

阶段验收：

- `build`、`lint`、测试全部通过。
- 同一天修改出生信息后，灵符与运势会立即更新。

### Phase B：打通可观测的增长闭环（1-2 周）

目标：知道用户从哪来、做了什么、有没有回流。

1. 把埋点从本地存储升级为真实可看报表
   - 轻量优先：Umami、PostHog 或自建简单接收端。
   - 事件至少包含：首次进入、生成结果、展开详情、换一批、商品点击、保存分享图、语言切换、回访。

2. 改造分享图回流路径
   - 分享图加二维码或明确链接。
   - 增加可复制的分享文案模板。
   - 区分“保存图片”和“复制文案”两类分享动作。

3. 建立基础漏斗指标
   - 安装/打开
   - 首次生成结果
   - 分享保存率
   - 商品点击率
   - 次日或 7 日回访

阶段验收：

- 能看到真实用户漏斗，而不只是本地调试数据。
- 每个分享动作都能回流到一个明确入口。

### Phase C：把变现链路做成真的（1-2 周）

目标：先验证“有人点、有人买”，再谈更重的商业化。

1. 把商品配置从代码中拆出来
   - 提出独立 JSON 或配置层。
   - 支持后续替换商品、排序、下线。

2. 接入真实推广链路
   - 中文优先选一个渠道先打通，不要同时做淘宝/京东/拼多多三套。
   - 如果联盟 API 成本高，先用自己的中转落地页做点击归因。

3. 补商品展示信息
   - 商品图片
   - 商品来源标识
   - 更明确的 CTA

阶段验收：

- 至少有一条可归因的商品点击链路。
- 能区分“看了商品”和“点了商品”。

### Phase D：渠道验证后再做平台决策（2-4 周）

目标：用数据决定是继续扩展，还是转向小程序。

1. 先做低成本分发实验
   - 完整 README
   - 安装说明
   - 对外截图
   - 发布包整理

2. 运营上做小实验，而不是先做大矩阵
   - 连续 2 周产出固定频率内容
   - 记录每条内容带来的访问、安装、分享、点击

3. 到这一步再决定是否做小程序
   - 如果内容点击高，但扩展安装差，优先立小程序。
   - 如果扩展留存和点击已经成立，继续做扩展迭代即可。

阶段验收：

- 能用数据回答“扩展是不是错的产品形态”。

## 五、明确不做的事

在下面这些前提成立之前，不建议投入：

- 不做付费解锁全符集。
- 不做定制灵符支付导出。
- 不做 NFT / 数字藏品。
- 不做大规模多平台内容矩阵自动化。
- 不同时接三家电商联盟。

前提只有两个：

1. 用户会回来。
2. 用户会点分享或商品。

这两个没成立，后面的商业化都是伪命题。

---

## 六、建议的下一步执行顺序

本周只做三件事：

1. 修缓存正确性 + 补测试 + 清 lint。
2. 上真实可见的埋点。
3. 给分享图加回流入口。

这三件做完，再决定是继续做真实联盟转化，还是提前启动小程序。
